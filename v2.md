## 1. Per-source / per-agent prompt fragment ("Bajour as 'we'")

**Status:** ❌ NOT addressed in v1 of external-drafts work.

**Problem.** When Dorfkönig surfaces a unit whose source is `bajour.ch`, the
draft refers to it like any other third-party source ("wie bajour.ch
berichtet"). But Bajour is *us* — the editorial voice should switch to "wir
berichten" / "[Journalist Name] hat herausgefunden" when the source domain
matches a partnered publisher. Today there is no per-source prompt fragment
mechanism; `compose-draft.ts` builds one prompt for all units regardless of
source.

**Shape of the fix.**

- Schema: add `scouts.system_prompt_fragment TEXT NULL` and/or a small
  `source_voice_overrides` table keyed by domain (`bajour.ch` → "treat as own
  publication, use 'wir' / first-person voice; cite by author name").
- Selection step: pre-compute which selected units come from a domain that has
  a voice override, attach that fragment to the prompt context.
- Compose prompt: append the fragment(s) right after `layer1` so the LLM gets
  per-bullet guidance like "this bullet's source is bajour.ch — use first-person
  voice".
- Validators: no change (validators are domain-agnostic).

**Estimated cost:** one migration (~20 lines), one prompt-builder param, one
join in the auto-draft selection step. ~80 LOC. Single-file PR plausible.

**Why it's separate.** Touches a different concern (source identity) and a
different schema (scouts table, not bajour_drafts). Bundling with external
drafts would have created unrelated review surface and a second migration.

## 2. Source-grounding / hallucination check

**Status:** ❌ NOT addressed.

**Problem.** Concrete case from 24. Apr. feedback: Dorfkönig wrote
"die Veranstaltung im Neuen Theater endet um 21 Uhr" — but the source page
never stated an end time. The model invented a plausible-sounding fact that
wasn't in the unit text. This is a real grounding failure, distinct from
factual errors in the source itself.

**Shape of the fix (one of):**

- **Compose-time validator.** After bullet generation, fetch the bullet's
  cited URL, embed both the page content and the bullet, run a high-threshold
  similarity check on each numeric/temporal claim. If a number/time appears in
  the bullet but cannot be found in the source page text, fail validation and
  drop the offending detail. This is the cheap, deterministic option but only
  catches numbers/times.
- **LLM judge step.** Send each generated bullet + its source unit's
  `context_excerpt` to a separate LLM call: "Does the bullet make any factual
  claim that is NOT supported by the excerpt? Return a list of unsupported
  claims." Repair or drop offending bullets. Higher cost (one extra call per
  bullet) but catches semantic hallucinations.
- **Retrieval-augmented compose.** Pass the full source `context_excerpt` for
  each unit into the compose prompt with strict "use ONLY facts present in the
  excerpt" framing. Easiest to ship but relies on prompt discipline.

**Recommendation:** start with the LLM judge step; it's the smallest change
that catches the 21-Uhr-class bug and degrades gracefully (a missed catch
still ships the draft).

**Estimated cost:** one new shared module (`_shared/draft-grounding.ts`),
~150 LOC, plus an OpenRouter call per bullet at compose time. Latency adds
~2–3s to auto-draft. Worth it given the editorial cost of hallucinations.
