Why now

The work is distributed, but the context is not.

Remote coordination

Important context stays in Slack threads, calls, and individual memory.

Repeated support

The same CMS and setup questions reappear without compounding into reusable answers.

Manual onboarding

Media setup needs too many handoffs, checks, and undocumented decisions.

Developer friction

Linear issues need richer briefs: product context, CMS state, media support history, and code pointers.

Core bet

AI only works when organizational knowledge becomes explicit, structured, and maintained.

The first product is not a chatbot. It is an operating system for memory, work, and safe action.

Support workflow

Support is a form plus a skill, not a free-form bot.

Slack stays conversational, but Hermes should push every support thread through a structured workflow before answering.

1. Intake form

  • media name
  • CMS URL / environment
  • problem category
  • urgency and screenshots

2. Skill preflight

  • classify request
  • check KB pattern
  • detect sensitive data
  • decide escalation path

3. Live check

  • CMS MCP read tools
  • no writes by default
  • Linear issue if engineering needed

4. Human gate

  • draft Slack reply
  • draft Linear issue
  • propose KB update
  • human sends/approves
Onboarding workflow

Onboarding needs gates, not one big prompt.

The form captures state, the skill enforces gates, scripts do local deterministic setup, and CMS MCP handles authenticated checks and approved changes.

Form

  • stakeholders
  • domain and branding
  • payment provider choice
  • subscriptions and plans

Skill gates

  • G0 access and identity
  • G1 design.md
  • G2 CMS setup state
  • G3 payments approval

Local script

  • create media folder
  • generate templates
  • validate required docs
  • never store secrets

CMS MCP

  • read setup status
  • dry-run proposed changes
  • execute only after approval
  • write audit log
Sensitive data boundary

Sensitive setup data should not become KB content.

Allowed in KB

Setup status, decisions, public URLs, owners, blockers, design notes, and non-secret configuration summaries.

Allowed in form

Structured business inputs. The form must label sensitive fields and route them to the correct system.

Allowed in CMS

Payment details, credentials, provider setup, and live configuration belong in authenticated CMS/admin systems.

Allowed in scripts

Local scaffolding and validation only. Scripts can create files and check completeness; they must not transmit secrets.

60-day rollout

Move in four two-week increments.

Weeks
1-2

Foundation

  • Create KB vault and media page templates.
  • Define llms.txt, Linear templates, and internal developer Slack ingestion.
  • Draft support-triage, developer-briefing, and kb-ingest-slack-linear.
Weeks
3-4

First inboxes

  • Open one support/onboarding channel.
  • Hermes drafts support replies and Linear issues with human approval.
  • Finish CMS MCP read-only diagnostics and start weekly KB maintenance.
Weeks
5-6

Media context

  • Add per-media setup knowledge, design.md, stakeholders, and funding context.
  • Hermes generates setup reports from CMS MCP.
  • Add cms-mcp-operator and payment-setup skills.
Weeks
7-8

Controlled action

  • Add narrow human-approved CMS MCP write tools.
  • Pilot support/onboarding copilot with one media organization.
  • Review Hindsight only as optional Hermes memory, not as the shared KB.
Governance

Human-gated by default.

No PR merges

Agents can draft and prepare. Humans decide when to merge.

No blind CMS writes

Mutations require dry-run, approval, scope, and audit logs.

Access boundaries

Public docs, internal KB, restricted CMS state, human-approved actions.

Support tone

Media-facing answers stay practical and avoid implementation leakage.
